Workflows

Research, journal & conference management over one grammar

The operating system stopped mattering. When the host is minimal and everything above it is a reproducible, portable capability, Linux vs Windows vs macOS is an implementation detail — not a constraint on the work.

The premise

Most researchers still wire their tooling to a machine: a specific laptop, a specific OS, a specific set of installed apps. That coupling is the reason a workflow that runs in the lab dies on a field laptop, a collaborator's desktop, or an air-gapped reviewer's machine. It does not have to be that way. Two ideas, taken seriously, remove the coupling entirely:

1 · One workflow grammar

Every research, publication, journal, and conference process decomposes into the same small set of verbs. Once you see the grammar, "journal submission" and "conference submission" stop being bespoke procedures and become parameterised instances of one pattern.

2 · A minimal host

Keep the operating system small, stable, and boring. Put every capability that actually does work — the toolchains, the models, the publishing pipeline — into reproducible units that run anywhere. The host becomes an execution substrate, not a development machine.

The rest of this page makes both concrete: the grammar first, then journal and conference management as instances of it, then the argument for why the OS is immaterial — and what that buys a researcher in a place with no reliable internet.

The universal workflow grammar

Industry, research, government, education — every domain's processes reduce to twelve verbs. This is not a metaphor; it is the operational vocabulary that the publishing, journal, and conference flows below are built from.

Observe
sense the current state
Acquire
obtain inputs, rights, funding
Create
produce the artifact
Transform
convert between forms
Validate
check against a standard
Approve
gate on a decision
Publish
release to an audience
Operate
run the live thing
Maintain
keep it correct over time
Archive
preserve immutably
Recover
restore from a known state
Retire
end-of-life with provenance

Compliance is not a thirteenth verb — it is a property attached to the verbs. Anonymisation, consent (IRB/IACUC/IEC), privacy law (GDPR/HIPAA), ethics review, export control, records management, and legal hold are checkpoints embedded inside Acquire, Transform, Publish, and Archive. A workflow is compliant when those checkpoints are verifiable nodes in the graph, not a PDF someone signed afterwards.

Journal workflow management

A journal submission is a fixed path through the grammar. Naming the stages is what makes the process trackable, automatable where an API exists, and honestly manual where one does not.

Before submission

Observe → Validate
Journal discovery and fit-scoring: does the venue's scope, audience, and impact match the work? Formatting automation to the target's template. Cover-letter generation.

Submission

Publish → Approve
Package the manuscript, figures, and supplementary material. Most journals (OJS, ScholarOne, Editorial Manager) expose no public API — so this stage is a tracked manual hand-off, fed by a complete, reviewer-ready bundle.

Under review

Maintain
Status tracking (submitted → under review → revisions → accepted). Reviewer-response management: every comment mapped to a change, with a revision trail. Supplementary-material versioning.

Acceptance

Archive
Camera-ready, then deposit. The accepted version is timestamped and assigned a DOI so the record is permanent and citable independent of the publisher.

The load-bearing honesty: where a portal has no API, the workflow does not pretend to automate it. It produces a self-contained, checksummed bundle — manifest, all metadata formats, a cryptographic timestamp — and hands that to the manual step. The tracking layer (a small status record per paper) is what turns "I think it's still under review" into a fact.

Conference workflow management

Same grammar, different parameters. Conferences add a deadline economy and a presentation tail that journals lack.

Discovery

Observe
Call-for-papers discovery and deadline tracking across venues. Scope-matching against the work in flight.

Submission

Transform → Publish
Anonymised version generation for double-blind review, submission packaging, and the camera-ready workflow. Most conference systems (EasyChair, HotCRP) are, again, manual portals fed by a complete bundle.

Review & response

Maintain
Reviewer-response management and rebuttal tracking under a hard clock.

Presentation

Operate → Archive
Presentation and poster generation, poster variants, travel planning. The talk and slides are deposited alongside the paper so the artifact outlives the event.

The same bundle feeds everything downstream

Research → development → experiment → DOI is the spine. Once an artifact is publication-worthy, a single canonical metadata record drives every output format — Zenodo deposit, DataCite, codemeta, CITATION.cff — and a single result with a DOI becomes the trigger for the announcement, the community post, and the educational follow-on. One record in; many channels out. That is the only point where the "experiment" and "publication" workflows need to touch.

Why the operating system is immaterial

Here is the part that surprises people. Every workflow above runs identically on Linux, Windows, macOS, a Raspberry Pi in a village classroom, or a cloud VM — because none of the work lives in the operating system. The discipline that makes this true is simple and old:

The host stays minimal

The operating system's job is reduced to four things: hardware enablement, a container runtime, networking, and a cryptographic identity. It is small, stable, and slowly-changing. Nothing project-specific is installed on it. It is replaceable and disposable.

Capabilities live above it

Every toolchain, model, language runtime, and publishing pipeline lives in a reproducible unit — a container or an equivalent declarative environment. These are instantiated on demand and discarded without perturbing the host. The capability is portable; the host is interchangeable.

State lives in artifacts, not machines

Code, metadata, contracts, inventories, and provenance live in repositories and registries — not in one laptop's home directory. Any machine that can read the repository can reconstruct the full working state. The conversation that built it is a bootstrap, not the memory.

Once those three hold, the OS underneath is a detail you stop thinking about — the same way you stop thinking about which brand of electricity meter is on the wall. Computing becomes infrastructure. The question "but does it run on my OS?" dissolves, because the answer is always yes: the host only has to run the runtime, and the runtime runs everywhere.

The deployment continuum — including where the internet is bad

This is not a cloud argument. The same capability runs across a continuum, with graceful degradation and seamless synchronisation when connectivity returns. AGI approaching does not mean every remote corner has good internet — so the base layers are common and always-present, and the heavier upper layers arrive just-in-time when a link is available, or ship pre-staged when it is not.

Local

A field researcher on an intermittent connection runs the offline profile: the common base layer is already on the device, the work proceeds, and results sync upward when a link appears. A lab with servers runs the private profile. A burst of heavy compute borrows the public tier and releases it. The workflow does not change — only which tier it lands on. That is what "seamless private to hybrid to public" actually means in practice: one definition, many places it can execute, no rewrite between them.

What this means for you, concretely

If you are setting up to work this way, the order is: (1) get the universal base layer running on whatever host you have — the runtime, nothing more; (2) pull the capability units you need for your domain on demand; (3) keep your state in a repository so any machine can resume it. After that, your choice of operating system is yours to make on comfort and cost, not on capability — and your workflow will outlive the machine you started it on.

The base layers are common to everyone. The upper layers are yours, on demand or pre-staged. That is the whole arrangement.
